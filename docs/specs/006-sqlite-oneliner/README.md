# SPEC-006: SQLite Default + One-Liner Install + Agent Join

**Status**: Draft
**Version**: 1.0
**Created**: 2026-05-08
**Priority**: Critical (adoption blocker)
**Compliance**: Must pass [SPEC-000](../000-api-compliance/README.md) at every phase

## Overview

Transform O3K from a PostgreSQL-only system requiring Docker Compose setup into a K3s-style zero-dependency single binary that starts with one command and stores state in embedded SQLite. Agent nodes join the cluster with a single `--join` flag. PostgreSQL becomes the opt-in upgrade path for multi-server HA.

**User experience goal:**

```bash
# Server (one command, zero deps):
curl -sfL https://get.o3k.io | sh -
# OpenStack running at localhost:35357 in under 60 seconds

# Add compute node:
o3k agent --server https://10.0.0.1:6443 --token <auto-generated>
```

## Goals

1. **Zero-dependency single-node**: `o3k server` starts everything with embedded SQLite — no PostgreSQL, no Docker, no config file required
2. **One-liner install**: Download script detects OS/arch, downloads binary, installs systemd unit, starts service
3. **Agent join**: `o3k agent --server <url> --token <token>` joins a compute node with zero config
4. **PostgreSQL upgrade path**: `o3k server --datastore postgres://...` switches to PostgreSQL for multi-server HA
5. **Data migration**: `o3k migrate-datastore --from sqlite --to postgres://...` moves data between backends
6. **K3s UX parity**: Anyone familiar with K3s should feel at home

## Non-Goals

- SQLite multi-server (explicitly impossible — single-writer limitation)
- Distributed SQLite (litestream, rqlite — adds complexity counter to the goal)
- Removing PostgreSQL support (it remains the HA path)
- Migration from PostgreSQL back to SQLite

### Key Design Decision: Query Compatibility Layer

**Problem**: 821 query sites use PostgreSQL `$1, $2, $3` placeholders. SQLite uses `?, ?, ?`.

**Solution**: The adapter transparently rewrites `$N` → `?` at the adapter boundary. Service code writes PostgreSQL-style queries unconditionally. The SQLite adapter does a single `regexp.ReplaceAll` before executing.

```go
// internal/database/sqlite_adapter.go
func (a *SQLiteAdapter) Exec(ctx context.Context, sql string, args ...any) (Result, error) {
    sql = rewritePlaceholders(sql) // $1,$2,$3 → ?,?,?
    sql = rewriteDialect(sql)      // ILIKE→LIKE, NOW()→CURRENT_TIMESTAMP, etc.
    return a.db.ExecContext(ctx, sql, args...)
}
```

This means **zero changes to the 821 query sites**. The adapter handles all dialect translation.

### SQL Dialect Translation Rules

| PostgreSQL | SQLite | Handled By |
|-----------|--------|------------|
| `$1, $2, $3` | `?, ?, ?` | Placeholder rewriter |
| `ILIKE` | `LIKE` (SQLite LIKE is case-insensitive for ASCII) | Dialect rewriter |
| `NOW()` | `CURRENT_TIMESTAMP` | Dialect rewriter |
| `::text`, `::int`, `::jsonb` | Remove cast (SQLite is typeless) | Dialect rewriter |
| `JSONB` column type | `TEXT` (store JSON as string) | Migration files |
| `gen_random_uuid()` | Generated in Go before INSERT | UUID helper |
| `FOR UPDATE SKIP LOCKED` | `BEGIN IMMEDIATE` (single writer) | Scheduler redesign |
| `EXTRACT(EPOCH FROM x)` | `strftime('%s', x)` | Dialect rewriter |
| `ON CONFLICT DO NOTHING` | Same syntax (supported since SQLite 3.24) | No change |
| `RETURNING` | Same syntax (supported since SQLite 3.35) | No change |
| `BOOLEAN` | `INTEGER` (0/1) | Migration files |

### Migration Strategy

Two sets of migration files:

```
migrations/
├── postgres/
│   ├── 001_initial_schema.up.sql
│   ├── 001_initial_schema.down.sql
│   └── ... (existing 71 migrations)
└── sqlite/
    ├── 001_initial_schema.up.sql   (converted from postgres)
    └── ... (same logical migrations, SQLite syntax)
```

**Conversion rules for migration files:**
- `UUID` → `TEXT`
- `JSONB` → `TEXT`
- `BOOLEAN` → `INTEGER`
- `DEFAULT gen_random_uuid()` → removed (Go generates UUIDs)
- `SERIAL` / `BIGSERIAL` → `INTEGER PRIMARY KEY AUTOINCREMENT`
- `NOW()` → `CURRENT_TIMESTAMP`

### Scheduler Redesign for SQLite

The task scheduler currently uses `FOR UPDATE SKIP LOCKED` which is PostgreSQL-only. For SQLite (single-server only), this simplifies to:

```go
// SQLite mode: single writer, no lock contention possible
func (w *Worker) claimTask(ctx context.Context) (*Task, error) {
    tx, _ := db.BeginTx(ctx, nil) // BEGIN IMMEDIATE in SQLite
    row := tx.QueryRow(`SELECT id, ... FROM tasks WHERE status = 'pending' ORDER BY created_at LIMIT 1`)
    // ... scan ...
    tx.Exec(`UPDATE tasks SET status = 'running', claimed_by = ? WHERE id = ?`, ...)
    tx.Commit()
    return task, nil
}
```

For PostgreSQL mode, the existing `SKIP LOCKED` implementation remains unchanged.

## One-Liner Install Script

### `install.sh` (hosted at get.o3k.io)

```bash
#!/bin/sh
set -e

# Detect OS and architecture
OS=$(uname -s | tr '[:upper:]' '[:lower:]')
ARCH=$(uname -m)
case $ARCH in
    x86_64) ARCH="amd64" ;;
    aarch64|arm64) ARCH="arm64" ;;
    *) echo "Unsupported: $ARCH"; exit 1 ;;
esac

# Download binary
VERSION=${O3K_VERSION:-latest}
URL="https://github.com/cobaltcore-dev/o3k/releases/download/${VERSION}/o3k-${OS}-${ARCH}"
curl -sfL "$URL" -o /usr/local/bin/o3k
chmod +x /usr/local/bin/o3k

# Create data directory
mkdir -p /var/lib/o3k

# Install systemd service (if systemd)
if command -v systemctl >/dev/null 2>&1; then
    cat > /etc/systemd/system/o3k.service <<EOF
[Unit]
Description=O3K OpenStack Server
After=network.target

[Service]
ExecStart=/usr/local/bin/o3k server
Restart=always
RestartSec=5
LimitNOFILE=65535

[Install]
WantedBy=multi-user.target
EOF
    systemctl daemon-reload
    systemctl enable --now o3k
fi

# Wait for ready
echo "Waiting for O3K to start..."
for i in $(seq 1 30); do
    if curl -sf http://localhost:35357/v3 >/dev/null 2>&1; then
        echo "O3K is ready!"
        echo ""
        echo "  API:     http://localhost:35357/v3"
        echo "  User:    admin"
        echo "  Pass:    $(cat /var/lib/o3k/initial-password)"
        echo ""
        echo "  Add nodes: o3k agent --server https://$(hostname -I | awk '{print $1}'):6443 --token $(cat /var/lib/o3k/agent-token)"
        exit 0
    fi
    sleep 1
done
echo "Timed out waiting for O3K"
exit 1
```

### Server Startup (no config)

When `o3k server` starts with no `--datastore` flag:
1. Creates `/var/lib/o3k/` data directory
2. Creates SQLite database at `/var/lib/o3k/db/state.db`
3. Runs all SQLite migrations
4. Generates random JWT secret → `/var/lib/o3k/jwt-secret`
5. Generates random admin password → `/var/lib/o3k/initial-password`
6. Generates agent join token → `/var/lib/o3k/agent-token`
7. Seeds admin user + default project
8. Starts all 5 services + agent API endpoint on :6443
9. Writes readiness marker

### Agent Join

```bash
o3k agent --server https://10.0.0.1:6443 --token abc123
```

The agent:
1. Validates token against server's `/v1/agent/register` endpoint
2. Receives its node-id and configuration
3. Establishes gRPC tunnel to server (existing `internal/tunnel/`)
4. Reports compute capabilities (vCPUs, RAM, storage)
5. Starts executor loop, waiting for tasks from server

**Token validation**: Server stores a hash of the agent token. Agents present the raw token on first connect. Server validates, issues a client certificate (mTLS for subsequent connections).

## Implementation Phases

### Phase 1: Database Abstraction (2 weeks)

**Goal**: Decouple service code from pgx concrete types.

1. Redesign `DBIF` interface to use dialect-neutral types:
   ```go
   type Result interface {
       RowsAffected() int64
   }
   type Row interface {
       Scan(dest ...any) error
   }
   type Rows interface {
       Next() bool
       Scan(dest ...any) error
       Close()
       Err() error
   }
   type Tx interface {
       Exec(ctx context.Context, sql string, args ...any) (Result, error)
       QueryRow(ctx context.Context, sql string, args ...any) Row
       Query(ctx context.Context, sql string, args ...any) (Rows, error)
       Commit() error
       Rollback() error
   }
   ```

2. Create PostgreSQL adapter wrapping pgxpool (preserves all current behavior)
3. Update all 59 service files to use new interface return types
4. Replace 132 `pgx.ErrNoRows` references with `database.ErrNoRows` sentinel
5. All existing tests must still pass

### Phase 2: SQLite Adapter (2 weeks)

**Goal**: Working SQLite backend passing all unit tests.

1. Implement SQLite adapter using `modernc.org/sqlite` (pure Go, no CGO)
2. Implement placeholder rewriter (`$N` → `?`)
3. Implement dialect rewriter (ILIKE, NOW(), casts, EXTRACT)
4. Convert all 71 migration files to SQLite dialect → `migrations/sqlite/`
5. Implement UUID generation in Go (for tables using `gen_random_uuid()`)
6. Redesign scheduler task claiming for single-writer mode
7. All unit tests pass with SQLite backend
8. Integration test: full CRUD lifecycle on SQLite

### Phase 3: Zero-Config Server Mode (1 week)

**Goal**: `o3k server` works with no arguments.

1. Implement `--datastore` flag: `sqlite:///var/lib/o3k/db/state.db` (default) or `postgres://...`
2. Auto-create data directory structure
3. Auto-generate JWT secret, admin password, agent token on first start
4. Auto-run migrations on startup
5. Write readiness file after startup
6. Implement `o3k server --reset` for clean slate

### Phase 4: Agent Join Protocol (1 week)

**Goal**: `o3k agent --server <url> --token <token>` adds a compute node.

1. Server: `/v1/agent/register` endpoint (validates token, returns config)
2. Server: `/v1/agent/connect` WebSocket/gRPC endpoint (task dispatch)
3. Agent: `--server` and `--token` flags
4. Agent: Auto-reconnect with exponential backoff
5. Agent: Report capabilities on connect
6. Agent: Execute VM lifecycle tasks from server
7. Agent needs NO database — purely stateless executor

### Phase 5: Install Script + Release Pipeline (1 week)

**Goal**: `curl -sfL https://get.o3k.io | sh -` installs and starts.

1. GitHub Actions: Build multi-arch binaries (linux/amd64, linux/arm64)
2. GitHub Releases: goreleaser config
3. Install script: download, verify checksum, install binary, setup systemd
4. Landing page: get.o3k.io (static page hosted on GitHub Pages)
5. Smoke test: CI job that runs the install script on fresh Ubuntu

### Phase 6: Data Migration Tool (1 week)

**Goal**: `o3k migrate-datastore --from sqlite --to postgres://...`

1. Read all tables from source
2. Write all rows to destination (batch inserts)
3. Verify row counts match
4. Update config to point to new datastore
5. Graceful: can run while server is stopped

## Technical Decisions

### Why `modernc.org/sqlite` (Pure Go)?

| Library | CGO Required | Cross-compile | Single Binary | Performance |
|---------|-------------|---------------|---------------|-------------|
| `mattn/go-sqlite3` | Yes | Hard | No (needs libsqlite3) | Fast |
| `modernc.org/sqlite` | No | Easy | Yes | ~80% of mattn |
| `crawshaw.io/sqlite` | Yes | Hard | No | Fast |

K3s uses `mattn/go-sqlite3` via kine, but K3s doesn't need single-binary cross-compilation (they build with CGO in CI). For O3K's "download one binary" promise, `modernc.org/sqlite` is the only viable choice.

Performance delta: ~20% slower than CGO version. For O3K's workload (mostly reads, low write contention on single node), this is acceptable. PostgreSQL remains the performance path.

### Why Not an ORM?

Adding GORM/ent/sqlc would solve the dialect problem but:
- Requires rewriting all 821 query sites (same effort as our adapter approach)
- Adds a heavy dependency
- Reduces query control (important for performance tuning)
- The adapter approach is simpler: one `rewritePlaceholders` function handles 90% of the translation

### SQLite Limitations (Accepted)

| Limitation | Impact | Mitigation |
|-----------|--------|------------|
| Single writer | No multi-server HA | PostgreSQL for HA |
| No `FOR UPDATE SKIP LOCKED` | Scheduler redesign needed | `BEGIN IMMEDIATE` sufficient for single-server |
| ~100 concurrent connections practical max | Fine for single-server | PostgreSQL for scale |
| File-based (no network access) | Agents can't query directly | Agents are stateless — they call server API |
| Max ~1TB practical | Fine for most deployments | PostgreSQL for large scale |
| No `ILIKE` | Case-insensitive search different | SQLite LIKE is already case-insensitive for ASCII |

### Agent Architecture (Stateless)

Agents have NO database. They are pure executors:
- Receive tasks via gRPC from server
- Execute (libvirt, netlink, storage operations)
- Report results back to server
- Server updates the database

This means SQLite's single-server limitation doesn't affect agent scalability — you can have 100 agents connected to one server with SQLite.
